Create `src/app/api/library/delete/route.ts` with a named `DELETE` export.

The handler must:

1. Authenticate using the same service role supabase client + Bearer token pattern as Task 1. Return 401 with `{ error: "请先登录", code: "UNAUTHORIZED" }` if not authenticated.

2. Parse request body: `const { id } = await request.json()`. If `id` is missing or not a string, return 400 with `{ error: "缺少知识条目 id", code: "INVALID_ID" }`.

3. Run a Drizzle delete:
   ```typescript
   const deleted = await db
     .delete(knowledgeItems)
     .where(and(eq(knowledgeItems.id, id), eq(knowledgeItems.userId, user.id)))
     .returning({ id: knowledgeItems.id });
   ```
   The `and(id match, userId match)` double-check prevents users from deleting other users' items. `review_state` will cascade-delete automatically because the database has a FK relationship — note: the schema has `knowledgeItemId` in reviewState but no explicit cascade defined in Drizzle; the executor must add a database-level CASCADE constraint or handle it by also deleting from reviewState before deleting from knowledgeItems. Since Drizzle schema does not define `references()` with cascade, delete from reviewState first:

   ```typescript
   // 先删除 review_state（无 CASCADE 定义）
   await db.delete(reviewState).where(eq(reviewState.knowledgeItemId, id));
   // 再删除 knowledge_item（含所有者校验）
   const deleted = await db
     .delete(knowledgeItems)
     .where(and(eq(knowledgeItems.id, id), eq(knowledgeItems.userId, user.id)))
     .returning({ id: knowledgeItems.id });
   ```

4. If `deleted.length === 0`, return 404 with `{ error: "条目不存在或无权删除", code: "NOT_FOUND" }`.

5. Return `NextResponse.json({ deleted: deleted[0].id })` status 200.

6. Wrap entire handler in try/catch. On error: `console.error("删除知识条目错误:", error)` + return `{ error: "删除失败", code: "INTERNAL_ERROR" }` status 500.
